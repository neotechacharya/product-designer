CallHippo Dashboard Redesign

How I redesigned the CallHippo dashboard to give sales teams instant visibility into call performance — without navigating across multiple screens

Issues & Design Goals

The original dashboard had several usability challenges. Each issue directly shaped a design strategy and informed the final solution.

Issues With the Existing Dashboard

Issue 01

Lack of Clear Data Hierarchy

Metrics were presented without proper prioritization, making it difficult to identify the most important insights.

Issue 02

Limited Data Visualization

The system relied on basic charts that did not clearly represent trends or patterns — making performance invisible.

Issue 03

No Dedicated Analytics Section

Users had to navigate multiple areas to gather meaningful insights — there was no structured path from question to answer.

Issue 04

Poor Comparative Insights

It was difficult to compare inbound vs outbound calls, call outcomes, and performance trends — everything was flat and undifferentiated.

Design Strategy — Three Major Improvements

Create a Clear Analytics Structure

Introduce a dedicated Call Analytics module that organizes data into logical, role-relevant sections.

Improve Data Visualization

Use stacked charts, segmented metrics, and gauges to help users quickly understand patterns — not just read numbers.

Simplify the Dashboard

Reduce cognitive load by showing high-value metrics first and surfacing deeper insights only in the analytics module.

Problem

Sales managers were flying blind — basic metrics existed but no one could act on them fast enough

CallHippo is a cloud-based business phone system used by sales and support teams to manage inbound and outbound calls globally. As the product evolved and customer data increased, the existing dashboard struggled to communicate key call insights effectively.

Users had difficulty understanding call performance, identifying missed opportunities, and tracking agent activity. The typical workflow meant navigating across multiple screens, correlating data manually, and still walking away without a clear picture of what was actually happening.

The result: sales managers and operations teams were unable to quickly understand call performance or make data-driven decisions. Missed calls went unanalyzed. Agent performance trends stayed invisible. The system produced data without producing insight.

Why This Problem? — Auditing the Existing Dashboard

Before moving to solutions, I analyzed feedback from sales managers, call center supervisors, and support team leaders who used CallHippo daily. Patterns across roles were consistent and pointed to the same root cause: data existed, but it wasn't organized in a way that enabled decisions.

Research — Understanding Each Role's Real Pain

I analyzed feedback from users across three job functions — sales managers, call center supervisors, and support team leaders. The patterns were consistent and pointed to the same structural gap: the dashboard displayed data but was never designed to support decisions.

Insight 01

Poor information hierarchy made every insight require effort to find.

Metrics were presented without proper prioritization. The most important signals — missed calls, talk time, agent performance — were buried alongside low-value data.

Insight 02

Managers needed inbound and outbound separated — not aggregated.

Sales teams and support teams had fundamentally different questions. One view that served both ended up serving neither — each role was constantly filtering out data irrelevant to them.

Insight 03

Missed calls were a silent revenue leak with no tooling to diagnose them.

Teams knew they were missing calls. They had no way to identify whether the cause was staffing, routing, IVR design, or business hours — so the same issues recurred.

Information Architecture

Before designing any screen, I mapped how the two key roles would move through the redesigned system — from the main dashboard to the specific analytics views they each needed. This mapping revealed a two-tier structure: a simplified main dashboard for quick daily checks, and a dedicated analytics module for deeper investigation.

Sales Manager
Opens Dashboard → Scans Key Metrics → Opens Outgoing Analytics → Identifies Performance Trends ✓
Support Team Lead
Opens Dashboard → Views Missed Call Count → Opens Missed Call Analytics → Diagnoses Root Cause → Adjusts Routing / Staffing ✓

Key Design Decision from This Mapping

Mapping both paths together clarified that the main dashboard and the analytics module had different jobs. The dashboard had to answer "how are we doing?" in seconds. The analytics module had to answer "why?" — supporting investigation, not just monitoring. Designing these as two distinct but connected layers was the structural decision the entire redesign depended on.

The Problem it solves

The original dashboard presented metrics without prioritization — everything appeared equally important, which meant nothing stood out. Managers opening the dashboard at the start of the day couldn't immediately assess team activity or performance. Every insight required manual effort to extract.

Design Decisions

The redesigned dashboard surfaces six core metrics at the top level: Total Calls, Total Inbound, Total Outbound, Average Call Talk Time, Total Agents Called, and Best Performing Agent. These were selected based on what managers checked first in every session observed. Card-based layout with clear typography hierarchy lets the eye move quickly — number first, label second, context third. The goal was for a manager opening the dashboard to understand the day's call picture in under ten seconds, without clicking anything.

Impact

Managers now open the dashboard and have their first question answered before they've clicked anything. The dashboard functions as a daily briefing — a starting point that tells you whether everything is normal or whether something needs attention, and then gets out of the way.

The Problem it solves

The original system had no separation between inbound and outbound analytics — both were aggregated into a single view that was useful to neither role. Sales teams evaluating outbound campaign success and support leads monitoring inbound handling were both navigating the same undifferentiated data pool.

Design Decisions

The analytics module introduces dedicated views for incoming and outgoing calls — each with its own set of role-relevant metrics. Incoming analytics surfaces Total Incoming, Missed, Answered, Rejected, and Average Talk Time. Outgoing analytics surfaces Total Outgoing, Connected, Rejected, Not Answered, and Average Outgoing Talk Time. Stacked bar charts visualize call outcomes across time, making it immediately clear whether answered rates are improving or declining without requiring the user to calculate anything. Segmented metrics replace raw totals — so "2,340 inbound calls" becomes a breakdown that shows the full picture of what happened to those calls.

Impact

Sales managers can now evaluate outbound campaign performance trend-by-trend. Support leads can review inbound handling quality without filtering out irrelevant outbound data. Each role sees exactly what they need — and the visualization does the interpretation work for them.

The Problem it solves

Missed calls represent lost revenue and damaged customer relationships — but the previous dashboard only showed a total count. Teams knew they were missing calls; they had no way to understand why. Without root-cause data, every intervention was a guess. Routing changes, staffing adjustments, and IVR redesigns were made without evidence.

Design Decisions

The Missed Call Analytics view breaks down missed calls into five distinct reasons: not answered by agent, all agents unavailable, abandoned during welcome message, abandoned in IVR, and outside business hours. Each reason maps directly to a different operational fix — staffing, routing configuration, IVR design, or schedule adjustment. The visualization uses segmented horizontal bars to communicate relative proportions at a glance, making it immediately clear which reason is dominant without requiring mental arithmetic.

Impact

Teams can now see not just how many calls they missed, but why — and act on the right lever. A team missing calls because agents are unavailable needs a staffing fix. A team losing callers in the IVR needs a flow redesign. The data now points directly to the intervention.

Dedicated analytics modules for inbound, outbound, and missed call data

The redesigned dashboard transformed CallHippo from a data-display tool into a decision-support system. The main dashboard now functions as a daily briefing — six high-priority metrics that answer the most common managerial questions before anyone has clicked anything.

The Call Analytics module gave teams something they didn't have before: a structured path from question to answer. Missed call reasons, inbound trends, and outbound performance are now accessible without navigating across disconnected screens or building manual reports.

One outcome beyond the original scope: the analytics module became a tool for new manager onboarding. A manager joining the team could open the dashboard on day one and immediately understand how the team was performing — without needing to be taught which screens to navigate or how to interpret raw call logs. Structured visibility compresses onboarding in ways that documentation and training cannot.

What I Learned

  1. 1. Data-heavy interfaces require strong information hierarchy — not more data.

    The original dashboard wasn't missing data. It was missing structure. Adding more metrics to a poorly organized view would have made the problem worse. The redesign improved usability not by showing more, but by deciding what mattered most and designing around that priority.

  2. 2. Visualizations must communicate insight, not just represent data.

    A stacked bar chart communicates "how did answered vs. missed calls trend this week?" A raw table of daily totals requires the user to calculate the same thing manually. Choosing the right visualization format is the difference between a chart that creates insight and one that just transfers data.

  3. 3. Analytics tools should prioritize decision-making, not just data display.

    Every feature in the analytics module was designed around a specific decision it needed to support — not around the data that was available. "Why are calls being missed?" is a decision question. "How many calls were missed?" is just a number. Designing for the decision determines what data to surface and how to structure it.

  4. 4. Collaboration with engineering is critical — especially for data-driven products.

    Designing analytics features without engineering alignment risks building visualizations that can't be efficiently powered by the underlying data model. Working alongside engineers from the start meant the designs were grounded in what was technically feasible — and that tradeoffs were made intentionally rather than discovered at handoff.
